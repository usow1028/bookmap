# BOOKMAP
## 공공도서관 소장·대출 가능 도서 지도 검색 서비스 기획서

> 사용자가 주소를 입력하고 책을 검색하면, 가까운 공공도서관 중 해당 도서를 소장하고 있는 곳과 예상 이동 시간을 지도에서 바로 보여주는 경량 서비스

- 작성 기준일: 2026-03-27

## 1. 프로젝트 개요

Bookmap은 전국 공공도서관 관련 공공 데이터를 활용해, 사용자가 원하는 책을 가장 빠르게 찾고 이동할 수 있도록 돕는 지도형 검색 서비스다. 핵심은 ‘책 검색 결과’를 단순한 목록이 아니라 ‘지금 갈 수 있는 가까운 도서관’ 중심의 행동 가능한 정보로 재정렬하는 데 있다.

| 항목 | 내용 |
|---|---|
| 문서 목적 | Bookmap의 서비스 정의, MVP 범위, 데이터·API 구조, 운영 원칙, 초기 실행 로드맵을 정리한 실무형 기획서 |
| 핵심 포지션 | 기존 공공서비스 데이터를 다시 만들지 않고, 사용자가 실제로 움직일 수 있는 ‘지도 기반 탐색 경험’으로 경량화 |

## 2. 문제 정의

- 대한민국에는 수많은 국공립·공공도서관이 존재하지만, 도서관별 웹사이트와 검색 경험이 분산되어 있어 사용자가 전체를 한 번에 파악하기 어렵다.
- 같은 생활권 안에 여러 도서관이 있어도 ‘어디가 가장 가깝고, 어디에 내가 찾는 책이 있으며, 지금 가볼 가치가 있는지’를 한 화면에서 판단하기 어렵다.
- 기존 공공 데이터와 국가 단위 서비스는 존재하지만, 일반 이용자 관점에서는 지도 기반 즉시 탐색 경험이 충분히 매끄럽지 않다.

## 3. 서비스 목표

> 한 줄 정의: 주소를 입력하고 책을 검색하면, 가까운 공공도서관 중 그 책이 있는 곳을 지도에서 바로 보여주는 서비스.

| 목표 | 설명 | 판단 기준 |
|---|---|---|
| 탐색 시간 단축 | 책을 찾기 위해 여러 도서관 사이트를 오가는 시간을 줄인다. | 검색 1회로 지도 결과 노출 |
| 행동 가능성 강화 | 거리·이동시간·소장 여부를 함께 보여줘 사용자가 바로 움직일 수 있게 한다. | 지도 핀 + ETA + 상태 표시 |
| 초기 진입 장벽 최소화 | 회원가입 없이 주소 입력 또는 현재 위치 허용만으로 사용할 수 있게 한다. | 비회원 사용 가능 |

## 4. 핵심 사용자 시나리오

- 사용자는 현재 위치를 허용하거나 주소를 1회 입력한다.
- 검색창에 제목·저자·ISBN 중 하나를 입력한다.
- 시스템은 서지 후보를 정규화한 뒤, 해당 도서를 소장한 공공도서관 후보를 조회한다.
- 지도는 사용자의 위치 기준으로 즉시 줌인되며, 가까운 후보 도서관 핀을 표시한다.
- 도서관 카드에는 거리, 예상 이동시간, 소장 여부, 최근 수집 기준 대출 가능 여부, 도서관 바로가기 정보가 표시된다.

## 5. MVP 범위

| 구분 | 포함 | 제외 |
|---|---|---|
| 회원 시스템 | 없음. 주소 1회 입력 또는 위치 권한 사용 | 로그인, 프로필, 개인 서재 |
| 검색 | 제목·저자·ISBN 기반 도서 검색 | 추천, 리뷰, 커뮤니티 |
| 지도 | 도서관 핀, 거리, ETA, 상태 카드 | 복잡한 필터, 테마 지도 |
| 도서관 액션 | 홈페이지 이동, 길찾기 연결 | 예약·대출 자체 처리 |
| 알림 | 초기 미포함 | 입고 알림, 즐겨찾기 알림 |

## 6. 제품 포지셔닝

Bookmap은 ‘통합 도서관 플랫폼’이 아니라 ‘공공도서관 데이터의 행동형 인터페이스’다. 차별점은 데이터 보유가 아니라 경험 설계에 있다. 즉, 사용자가 지금 생활권 안에서 가장 빠르게 책을 찾고 이동할 수 있도록 검색 결과를 공간 맥락으로 재구성한다.

## 7. 화면 구성 제안

| 화면 | 목적 | 주요 요소 | 비고 |
|---|---|---|---|
| 홈 | 즉시 검색 | 주소 입력/현재 위치, 검색창 | 회원가입 없이 진입 |
| 결과 | 후보 비교 | 지도, 핀, 리스트 카드, ETA | 핵심 화면 |
| 도서관 상세 | 행동 유도 | 주소, 운영 정보, 홈페이지, 길찾기 | 외부 이동 허용 |
| 오류/빈 결과 | 이탈 방지 | 검색 수정 제안, 반경 확대 제안 | UX 안정화 |

## 8. 데이터 및 API 설계

초기 서비스는 국가 공공 데이터와 지도 API를 조합하는 구조로 설계한다. 핵심은 ‘도서 소장 후보 조회’와 ‘위치 기반 재정렬’의 결합이다.

| 구분 | 후보 소스 | 용도 | 비고 |
|---|---|---|---|
| 도서 검색/서지 | 도서관 정보나루 | 제목·저자·ISBN 기반 후보 정규화 | ISBN 중심 정합성 확보 |
| 소장 도서관 조회 | libSrchByBook | 특정 도서를 가진 도서관 목록 확보 | 핵심 API |
| 소장·대출 가능 여부 | bookExist | 도서관별 hasBook, loanAvailable 확인 | loanAvailable는 전날 기준 |
| 지도 표시 | 네이버 Maps 또는 Google Maps | 지도 렌더링, 핀 표시 | 국내 UX는 네이버 우선 검토 |
| 주소 변환 | Geocoding | 주소 → 좌표 변환 | 현재 위치 허용 시 생략 가능 |
| 이동시간 계산 | Directions / Routes | 상위 후보 ETA 계산 | 비용 절감을 위해 후보 수 제한 |

## 9. 추천 검색 로직

1. 사용자 위치 확보(현재 위치 또는 주소 입력)
2. 검색어 기반 서지 후보 정규화
3. ISBN 기준 소장 도서관 후보 조회
4. 후보 도서관 좌표와 사용자 좌표를 비교해 직선거리 1차 정렬
5. 상위 5~10개 후보에만 경로 API를 호출해 실제 ETA 계산
6. 거리·ETA·운영 여부·대출 가능 여부를 조합해 결과 재정렬
7. 지도 핀과 리스트 카드를 동시에 렌더링

이 방식은 모든 도서관 후보에 대해 매번 경로 API를 호출하지 않아도 되므로 속도와 비용을 동시에 관리할 수 있다.

## 10. 운영 원칙 및 문구 정책

| 원칙 | 적용 문구/방식 |
|---|---|
| 실시간처럼 표현하지 않기 | ‘실시간 대출 가능’ 대신 ‘최근 수집 기준 대출 가능’으로 표시 |
| 회원가입 최소화 | 초기 버전은 비회원 중심으로 설계 |
| 외부 시스템 연동 존중 | 예약·대출은 각 도서관 사이트 또는 기존 국가 서비스로 넘김 |
| 결과 신뢰도 표시 | 마지막 확인 기준, 운영 정보 기준일을 함께 노출 |

## 11. 기술 구조 제안

- 프론트엔드: React/Next.js 기반 단일 페이지 앱
- 백엔드: 경량 API 서버(Node.js 또는 Python FastAPI)
- 캐시: 도서관 마스터 데이터, 좌표 변환 결과, 인기 검색 결과 캐싱
- DB: PostgreSQL + PostGIS 또는 간단한 초기 버전이라면 PostgreSQL만 사용
- 지도 추상화 계층: 네이버/구글 중 하나로 시작하되, 추후 교체 가능하도록 래퍼 계층 유지

## 12. 최소 데이터 모델

| 테이블 | 핵심 필드 | 목적 | 비고 |
|---|---|---|---|
| libraries | library_id, name, address, lat, lng | 도서관 마스터 저장 | 운영시간·홈페이지 포함 |
| books | isbn13, title, author, publisher | 서지 후보 정규화 | 중복 검색 성능 개선 |
| holdings_cache | isbn13, library_id, has_book, loan_available, checked_at | API 응답 캐시 | 시점 정보 보존 |
| search_logs | query, resolved_isbn, region, clicked_library | 초기 분석 | 개인정보 최소 수집 |

## 13. 리스크와 대응

| 리스크 | 설명 | 대응 |
|---|---|---|
| 대출 가능 상태 시차 | loanAvailable가 전날 기준일 수 있음 | 문구 정책과 확인 시점 표시 |
| 경로 API 비용 | 모든 후보 ETA 계산 시 비용 증가 | 직선거리 1차 정렬 후 상위 후보만 계산 |
| 데이터 누락/지연 | 개별 도서관 반영 시차 가능성 | 공식 홈페이지 바로가기와 기준일 노출 |
| 과도한 기능 확장 | 알림, 추천, 커뮤니티로 초점 분산 | 초기에는 지도 검색 한 가지에 집중 |

## 14. 초기 KPI

- 검색 후 결과 노출까지 평균 시간
- 검색 대비 도서관 클릭률
- 길찾기 버튼 전환율
- 빈 결과 비율
- 사용자 1인당 평균 검색 횟수

## 15. 8주 실행 로드맵

| 주차 | 목표 | 산출물 |
|---|---|---|
| 1-2주 | API 검증 및 데이터 매핑 | ISBN 검색 흐름, 도서관 좌표/주소 정리 |
| 3-4주 | 핵심 검색·지도 UI 구축 | 주소 입력, 검색창, 결과 지도, 리스트 카드 |
| 5-6주 | ETA 계산 및 정렬 고도화 | 직선거리 + Directions 조합 로직 |
| 7주 | 문구 정책·오류 처리 보강 | 빈 결과, 기준일 표기, 외부 링크 흐름 |
| 8주 | 소규모 사용자 테스트 | MVP 릴리스 후보, KPI 대시보드 초안 |

## 16. 결론

Bookmap의 강점은 기능을 더하는 데 있지 않다. 오히려 이미 존재하는 공공도서관 데이터를 가장 직관적인 사용자 행동 흐름으로 재조립하는 데 있다. 초기 버전은 ‘회원가입 없는 검색 → 지도 줌인 → 가까운 대출 가능 후보 확인’만 정확히 구현하면 충분하다. 이 범위를 선명하게 유지할수록 제품의 정체성과 완성도가 높아진다.

## 부록. 공식 확인 자료

- 도서관 정보나루 Open API 개선 공지: `bookExist`의 `loanAvailable` 제공 및 `libSrchByBook` API 개발
- 도서관 정보나루 Open API 매뉴얼
- NAVER Cloud Maps: Dynamic Map, Geocoding, Directions 기능 및 무료 이용량
- Google Maps Routes API: Compute Routes / Compute Route Matrix 문서

> ※ 본 기획서는 2026-03-27 기준 공개 문서를 바탕으로 정리되었으며, 실제 개발 전에는 각 API의 최신 이용약관·요금·호출 제한을 재확인해야 한다.
