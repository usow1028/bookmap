# Bookmap 정보나루 API 개선 설계

## 1. 목적

`Bookmap`은 현재 정보나루의 핵심 3개 API만 사용해

- 도서 검색: `srchBooks`
- 소장 도서관 조회: `libSrchByBook`
- 도서관별 소장/대출 가능 여부: `bookExist`

를 처리하고 있다.

정보나루의 `Open API 활용방법` 기준으로 보면, 현재 구현은 "찾은 책을 어느 도서관이 갖고 있는가" 수준까지만 사용 중이며,

- 도서 상세/이용분석
- 추천도서
- 도서관 통합정보
- 지역/도서관별 인기대출
- 월별 키워드와 독서량 지표

는 아직 제품 경험에 반영되지 않았다.

이 문서는 정보나루 공식 활용 가이드와 실제 API 응답 확인을 바탕으로 `Bookmap`의 개선 및 보완 방향을 설계한다.

## 2. 공식 활용방법 기준 업무 분류

정보나루 `Open API 활용방법` 페이지에서 확인한 업무별 API는 아래처럼 제품 관점으로 묶는 것이 적절하다.

### 2.1 도서관/장서 조회 업무

- `libSrch`: 정보공개 참여 도서관 조회
- `itemSrch`: 도서관별 장서/대출 데이터 조회
- `libSrchByBook`: 특정 ISBN을 소장한 도서관 조회
- `bookExist`: 특정 도서관의 특정 ISBN 소장/대출 가능 여부
- `extends/libSrch`: 도서관 기본정보 + 운영시간 + 휴관일 + 대출반납 추이 + 신착도서 통합 조회

### 2.2 도서 검색/상세/분석 업무

- `srchBooks`: 도서 검색
- `srchDtlList`: 도서 상세 조회
- `keywordList`: 도서 키워드 목록
- `usageAnalysisList`: 도서별 이용분석

### 2.3 추천 업무

- `recommandList?type=mania`: 마니아 추천도서
- `recommandList?type=reader`: 다독자 추천도서

### 2.4 수요/인기/트렌드 업무

- `loanItemSrch`: 전체 인기대출도서
- `loanItemSrchByLib`: 도서관/지역별 인기대출도서
- `extends/loanItemSrchByLib`: 도서관별 인기대출도서 통합
- `hotTrend`: 대출 급상승 도서
- `monthlyKeywords`: 이달의 키워드
- `readQt`: 지역별 독서량/독서율
- `usageTrend`: 도서관별 대출반납추이

## 3. 현재 Bookmap 구현과 갭

현재 `bookmap`은 실사용 목적상 기본 흐름은 맞지만, 정보나루가 제공하는 "분석형 API"를 거의 쓰지 못하고 있다.

### 3.1 현재 강점

- ISBN 기준 소장 도서관 탐색 흐름이 단순하고 빠르다.
- `bookExist`를 붙여 실제 대출 가능 여부까지 보여준다.
- NAVER Directions 5와 결합해 "가까운 곳부터 행동 가능하게" 보여주는 UX 방향은 맞다.

### 3.2 현재 부족한 점

- 도서 메타데이터가 `srchBooks` 수준에 머물러 설명력과 판별력이 약하다.
- 도서관 정보가 `libSrchByBook`에 포함된 최소 필드만 사용되어 운영시간/휴관일/전화번호/장서규모를 보여주지 못한다.
- 검색 결과가 "어디 있나"까지만 끝나고 "왜 이 책을 추천하는가", "비슷한 책은 무엇인가", "해당 도서관 이용 특성은 어떤가"까지 이어지지 않는다.
- 정보나루가 제공하는 지역/연령/트렌드 분석 데이터를 제품의 사전 탐색 경험에 활용하지 못하고 있다.
- 일부 API는 도서관별 데이터 제공 편차가 있는데, 현재 제품에는 이 편차를 흡수하는 기능 가드 설계가 없다.

## 4. 실제 응답 확인으로 본 설계 포인트

이번 턴에서 실제 API 응답을 확인한 결과, 문서 설명만보다 더 중요한 구현 포인트가 드러났다.

### 4.1 `srchDtlList`

- `description`
- `publication_date`
- `bookImageURL`
- `loanInfo`

를 제공한다.

의미:

- 현재 `srchBooks`만으로는 비어 있는 책소개를 실제 상세정보로 채울 수 있다.
- 검색 후보 선택 후 상세 카드 품질을 즉시 올릴 수 있다.

### 4.2 `usageAnalysisList`

- `loanHistory`
- `keywords`
- `coLoanBooks`
- `maniaRecBooks`
- `readerRecBooks`

가 한 번에 내려온다.

의미:

- 별도 API 여러 개를 따로 붙이지 않아도 "이 책을 읽은 사람들은 이런 책도 읽었다"와 "핵심 키워드"를 바로 만들 수 있다.
- 도서 상세 패널과 추천 패널을 단일 분석 응답으로 구성할 수 있다.

### 4.3 `extends/libSrch`

- `libInfo`
- `closed`
- `operatingTime`
- `BookCount`
- `loanByhours`
- `loanByDayofWeek`
- `newBooks`

를 포함한다.

의미:

- `libSrchByBook`로 찾은 도서관 목록에 `extends/libSrch`의 캐시를 조인하면, 실제 도서관 카드 품질을 크게 올릴 수 있다.
- 향후 도서관 상세 페이지를 mock 없이 복구할 수 있다.

### 4.4 `usageTrend`

- 일부 도서관은 `"해당 도서관은 정보가 제공되지 않습니다."` 오류를 반환한다.

의미:

- 분석형 보조 위젯은 "있으면 노출, 없으면 숨김" 방식의 feature gate가 필요하다.
- 제품은 "데이터 미제공"을 오류가 아니라 정상 상태로 처리해야 한다.

### 4.5 `extends/loanItemSrchByLib`

- 도서관 단위 인기대출도서를 연령대별 묶음으로 제공한다.

의미:

- 도서관 카드/상세 화면에서 "이 도서관은 어떤 독자층에 강한가"를 보여줄 수 있다.
- `Bookmap`을 단순 보유 검색기가 아니라 "근처에서 어떤 책 경험을 할 수 있는지"까지 확장할 수 있다.

## 5. 개선 설계

### 5.1 P0: 현재 검색 흐름 고도화

목표:

- 현재 검색 UX를 유지하면서 정보 밀도를 높인다.

설계:

1. `srchBooks`는 계속 검색 후보 조회에 사용한다.
2. 사용자가 ISBN을 선택하면 `srchDtlList`를 추가 호출해 상세 메타데이터를 보강한다.
3. `libSrchByBook` 결과를 받은 뒤 `bookExist`로 가용성을 조회한다.
4. 동시에 `extends/libSrch` 기반 도서관 캐시와 조인해 운영시간/휴관일/전화번호를 보여준다.

반영 화면:

- 선택 도서 카드
- 도서관 후보 카드
- 지도 옆 상세 패널

사용 API:

- `srchBooks`
- `srchDtlList`
- `libSrchByBook`
- `bookExist`
- `extends/libSrch` 또는 `libSrch`

기대 효과:

- 현재 제품 정체성은 유지하면서 "설명력 없는 결과 카드" 문제를 가장 빠르게 해소할 수 있다.

### 5.2 P1: 도서 분석과 추천을 붙인 탐색 확장

목표:

- 사용자가 한 권을 찾고 끝나는 것이 아니라, 도서관 방문 전 대안 도서를 함께 탐색하게 만든다.

설계:

1. ISBN 선택 후 `usageAnalysisList`를 호출한다.
2. 응답에서 아래 블록을 분리해 노출한다.

- 대출 이력 추이
- 핵심 키워드
- 함께 대출된 책
- 마니아 추천도서
- 다독자 추천도서

3. 추천 도서 클릭 시 다시 `libSrchByBook -> bookExist` 흐름으로 연결한다.
4. 추천도서는 "현재 선택한 도서관이 함께 소장 중인지"를 추가 배지로 보여준다.

반영 화면:

- 검색 결과 우측 사이드 패널
- 선택 도서 하단 추천 영역

사용 API:

- `usageAnalysisList`
- 보조적으로 `recommandList`

기대 효과:

- 검색 1회로 끝나는 단발성 경험이 아니라 "탐색 체류"가 생긴다.
- 도서관 방문 가치가 높은 대체 도서를 사용자에게 제시할 수 있다.

### 5.3 P1: 실도서관 상세 페이지 복구

목표:

- 이전 mock 상세 페이지를 실데이터 기반으로 되살린다.

설계:

1. `app/libraries/[id]`를 다시 만들되 mock 없이 구현한다.
2. `libCode`를 기준으로 `extends/libSrch` 캐시에서 기본 정보를 조회한다.
3. 가능하면 `usageTrend`를 붙여 요일별/시간대별 대출반납 추이를 보여준다.
4. `extends/loanItemSrchByLib`를 붙여 해당 도서관 인기대출도서를 연령대별로 보여준다.
5. `usageTrend` 미제공 도서관은 차트 블록을 숨기고 안내 문구만 노출한다.

반영 화면:

- 도서관 상세

사용 API:

- `extends/libSrch`
- `usageTrend`
- `extends/loanItemSrchByLib`

기대 효과:

- `상세 보기`가 다시 제품 가치 있는 행동이 된다.
- 운영시간/휴관일/도서관 특성까지 확인 가능한 완성형 흐름이 된다.

### 5.4 P2: 지역 맥락 탐색 추가

목표:

- 사용자 검색 전 단계에서 지역 독서 분위기와 인기 흐름을 보여준다.

설계:

1. 사용자가 위치를 확정하면 지역 코드를 해석한다.
2. 지역 코드로 아래 데이터를 조회한다.

- `loanItemSrchByLib` 또는 `loanItemSrch`
- `hotTrend`
- `monthlyKeywords`
- `readQt`

3. 홈 화면 상단 또는 위치 선택 직후 아래를 보여준다.

- 내 지역 인기대출도서
- 최근 급상승 도서
- 이달의 키워드
- 지역 독서량/독서율

기대 효과:

- 검색 전에 "지금 이 동네에서 어떤 책이 움직이고 있는지"를 보여줄 수 있다.
- 사용자가 정확한 책 제목이 없어도 탐색을 시작할 수 있다.

## 6. 데이터 아키텍처 설계

### 6.1 API 어댑터 분리

현재는 `lib/data4library.ts`에 검색 흐름이 집중되어 있다.

개선안:

- `lib/data4library/books.ts`
- `lib/data4library/libraries.ts`
- `lib/data4library/analytics.ts`
- `lib/data4library/trends.ts`

로 나눈다.

이유:

- 업무별 API를 제품 기능과 같은 단위로 묶는 편이 유지보수에 유리하다.
- XML 파서와 DTO 변환 코드를 영역별로 나누면 누락 필드 대응이 쉬워진다.

### 6.2 조인 전략

권장 조인 순서:

1. `srchBooks` 또는 `srchDtlList`로 도서 기준 정보 확보
2. `libSrchByBook`으로 소장 도서관 후보 확보
3. `bookExist`로 가용성 보강
4. `extends/libSrch` 캐시로 도서관 운영정보 보강
5. `usageAnalysisList`로 도서 분석/추천 보강

### 6.3 캐시 전략

- `extends/libSrch`: 일 단위 캐시
- `libSrch`: 주 단위 또는 일 단위 캐시
- `srchDtlList`: ISBN 기준 장기 캐시
- `usageAnalysisList`: ISBN 기준 중기 캐시
- `loanItemSrchByLib`, `monthlyKeywords`, `hotTrend`, `readQt`: 시간 또는 일 단위 캐시
- `bookExist`: 짧은 TTL 유지

이유:

- 운영정보와 도서 상세는 자주 바뀌지 않는다.
- 대출 가능 여부와 급상승/인기 지표는 상대적으로 자주 바뀐다.

## 7. 파서/품질 보완점

실응답 기준으로 아래 방어 설계가 필요하다.

- `publication_year`가 비정형 문자열일 수 있다.
- `bookImageURL`가 비어 있을 수 있다.
- `bookDtlUrl` 앞에 공백이 섞여 있을 수 있다.
- 연령/지역 분석 블록이 빈 태그일 수 있다.
- 일부 분석 API는 특정 도서관에서 아예 미제공일 수 있다.
- 같은 작품의 다른 권/판본이 추천 목록에 섞여 나올 수 있다.

권장 처리:

- 문자열 `trim`
- nullable 필드 허용
- 배열/단일 객체 혼합 응답 공통화
- "데이터 없음"과 "오류"를 분리
- 판본/권차 그룹핑 로직 추가

## 8. 제품 관점 보완점

### 8.1 현재 제품에서 바로 보완할 것

- 선택 도서 카드에 책소개와 출판일 추가
- 도서관 카드에 운영시간/휴관일/전화번호 추가
- 실도서관 상세 페이지 복구
- 추천도서와 함께 대출된 도서 패널 추가

### 8.2 검색 UX에서 보완할 것

- 같은 제목의 여러 판본을 작품 단위로 묶고 펼치기
- 검색어만 입력한 상태에서 인기/추천 탐색 진입점 제공
- 위치가 정해졌을 때만 추천도서의 "근처 소장" 여부 계산

### 8.3 운영 리스크 대응

- 정보나루 분석형 API 미제공 도서관 fallback UI
- API 응답 지연 시 카드 skeleton과 부분 로딩
- `bookExist` 실패 시 "소장 확인, 대출 가능 여부 미확인" 상태 분리

## 9. 우선순위 로드맵

### P0

- `srchDtlList` 연동
- `extends/libSrch` 캐시 조인
- 도서관 카드 품질 개선

### P1

- `usageAnalysisList` 기반 분석/추천 패널
- 실도서관 상세 페이지
- `usageTrend`/`extends/loanItemSrchByLib` 조건부 노출

### P2

- 지역 인기/급상승/키워드 홈 피드
- 독서량/독서율 기반 지역 탐색
- 연령대/관심군 기반 개인화 필터

## 10. 결론

정보나루의 공식 활용방법 기준으로 볼 때 `Bookmap`은 이미 핵심 검색 골격은 맞게 잡혀 있다.

다만 지금 상태는 "책이 어디에 있나"만 해결하는 수준이고, 정보나루가 제공하는 분석형 API를 붙이면 아래 단계까지 바로 확장할 수 있다.

- 왜 이 책을 고를 만한가
- 이 도서관은 어떤 곳인가
- 비슷한 책을 지금 어디서 함께 빌릴 수 있는가
- 이 지역에서는 지금 어떤 책 흐름이 움직이고 있는가

따라서 가장 좋은 다음 단계는 기능을 넓게 벌리기보다,

1. `srchDtlList`
2. `extends/libSrch`
3. `usageAnalysisList`

이 3축을 기존 검색 흐름에 조인해 제품 밀도를 올리는 것이다.
